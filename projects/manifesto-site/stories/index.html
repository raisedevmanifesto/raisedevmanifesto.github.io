These user stories are derived directly from our vision, defining the features that bring the RAISE methodology to life through the website. Each story is user-centered and focuses on delivering tangible value.

Core User Stories

The following stories guided the development of the RAISE Manifesto website, organized by user persona.

For Software Engineers

Understanding the Methodology

As a software engineer interested in AI-assisted development, I want to understand the core principles of RAISE methodology so that I can apply them in my own projects.

Acceptance Criteria:
  • Clear explanation of RAISE acronym and philosophy
  • Detailed breakdown of core principles
  • Accessible and scannable formatting
  • Connection to familiar software development concepts

Exploring Artifact Structure

As a developer new to RAISE, I want to see practical examples of RAISE artifacts so that I can create my own effective documentation.

Acceptance Criteria:
  • Examples of Vision Documents, Values statements, User Stories, etc.
  • Templates or structural guidelines for each artifact
  • Clear explanation of relationships between artifacts
  • Progressive disclosure of complexity

For Technical Leaders

Methodology Integration

As a technical leader, I want to understand how RAISE methodology integrates with existing development practices so that I can incorporate it gradually into my team's workflow.

Acceptance Criteria:
  • Comparison with other methodologies (Agile, etc.)
  • Guidance on incremental adoption
  • Recommendations for team roles and responsibilities
  • Discussion of common challenges and solutions

Evaluating Project Success

As a project manager, I want to know how to measure the success of RAISE implementation so that I can evaluate its impact on our development process.

Acceptance Criteria:
  • Suggested metrics for measuring methodology effectiveness
  • Warning signs of methodology drift
  • Case studies or testimonials from other teams
  • Guidelines for continuous improvement

For Non-Technical Stakeholders

Understanding Development Process

As a product owner without technical background, I want to understand how RAISE structures the development process so that I can participate meaningfully in project decisions.

Acceptance Criteria:
  • Non-technical explanation of RAISE workflow
  • Visual representations of process flow
  • Clarity about stakeholder touchpoints
  • Specific guidance for providing effective input

Contributing to Artifacts

As a domain expert, I want to contribute effectively to RAISE artifacts so that my knowledge is accurately captured in the development process.

Acceptance Criteria:
  • Templates for non-technical contributions
  • Guidelines for effective collaboration with technical team
  • Examples of successful cross-functional input
  • Clear language without technical jargon

For Site Visitors

Accessible Information

As a visitor with accessibility needs, I want to access all website content regardless of my abilities so that I can learn about RAISE methodology without barriers.

Acceptance Criteria:
  • WCAG 2.1 AA compliance
  • Screen reader friendly navigation
  • Keyboard accessible interactions
  • Sufficient color contrast
  • Responsive design for all devices

Finding Example Projects

As an interested visitor, I want to explore real-world examples of RAISE implementation so that I can understand its practical application.

Acceptance Criteria:
  • Gallery of example projects
  • Clear categorization and filtering options
  • Complete artifact sets for each example
  • Progressive journey from vision to implementation

These user stories were derived from our vision and values. They guide feature development and help evaluate whether we're delivering on our purpose.
